- ⏳ API versioning prefix (/v1/)                                                        P3d
  → All endpoints are at root (/class, /jar, /search, etc.); any breaking change needs a
    coordinated cutover with every client
  → Add /v1/ prefix now (costs nothing) and redirect bare paths for backwards compatibility

- ⏳ Embedded JAR extraction (nested JARs inside uploaded JARs)                         P3e
  → Current: JarBinariesController feeds the top-level JAR entries only
  → Target: recurse into nested JARs found at BOOT-INF/lib/*.jar (Spring Boot fat JARs),
    WEB-INF/lib/*.jar (WAR), and root /*.jar (Maven shade / Gradle shadow uber-JARs)
  → Each nested JAR is extracted as a separate logical unit under the same project/version,
    using the nested JAR filename as the unit identifier
  → Avoids re-uploading already-indexed library JARs: check by JAR content hash against
    a new postgres table (jar_content_hashes) before extracting; skip if already present
  → Prerequisite: P0e (streaming refactor) — Jimfs cannot efficiently handle nested ZipInputStream
  → Enables indexing of entire application distributions in a single upload

- ⏳ Config file extraction from within JARs                                            P3f
  → JARs routinely bundle application.yml, application.properties, *.properties resource files
  → During JAR upload (including nested JARs from P3e), detect config file entries by extension
    and path pattern (e.g. BOOT-INF/classes/*.yml, *.properties outside .class paths)
  → Feed detected entries to constant-extractor-config-file module (already exists as a module);
    results stored under the same project/version with source annotated as CONFIG_FILE
  → Means a single JAR upload produces both bytecode constants and config constants together
  → No new API surface needed — transparent to the caller

## V3 — Advanced / Long-term
## ════════════════════════════════════════════════════

- ⏳ Exception-flow bytecode analysis (catch block constant extraction)                  P4a
- ⏳ Solr dead-letter replay admin endpoint                                              P4b
  → SolrOutboxProcessor comment notes "future admin endpoint"; rows currently need manual SQL
  → POST /admin/solr-outbox/replay?id=... → move dead-letter row back to live outbox

- ⏳ [TENTATIVE] Replace Solr with Postgres-native search (pg_trgm + pgvector)          P4c
  → Rationale: Solr brings no meaningful advantage for short string constants; BM25 is tuned
    for long documents; constants benefit more from type-aware matching
  → UnitConstantsStore SPI already isolates Solr behind an interface — swap is clean
  → Postgres replacement: pg_trgm GIN index for fuzzy search, fuzzystrmatch levenshtein()
    for edit-distance, value_num numeric column for type-aware numeric similarity
  → Eliminates: SolrService, SolrOutboxProcessor, SolrOutboxPayload, solr_outbox table,
    P0d (Resilience4j), P2h (parsePairs inconsistency), P4b (dead-letter replay),
    Solr container in docker-compose + k8s, managed-schema.xml, SolrJ dependency
  → Prerequisites: all planned search features (P3c) must be built first so all
    query patterns are known before rewriting the store layer

- ⏳ Cross-component constant coupling analysis                                          P4d
  → Core value proposition: detect when the same (or near-same) constant value appears in
    multiple independently-indexed projects/versions — a hidden coupling that code review
    cannot reliably catch across component boundaries
  → New query: GET /search/shared?value=...  → returns all projects+versions that contain
    this exact constant value, ranked by frequency; string and numeric types handled separately
  → Subsumes P3a (similar constant detection): near-same numeric range handled here via
    Postgres value_num column or in-memory post-fetch; Solr MLT within-project approach dropped
    as too narrow and Solr-specific
  → Coupling strength signal: if two components share a constant AND that constant appears in
    conditional logic (branch predicate via ARITHMETIC_OPERAND or comparison context), flag as
    HIGH_RISK coupling — a one-sided change silently alters branching behaviour in the other
  → Condy coupling edge (requires P2k): if two components reference the same DynamicConstantDesc
    BSM (same owner + name + static args), record a BSM-level coupling edge — stronger than
    value equality because it means both depend on the same code to produce the constant
  → UI: "Shared across projects" indicator in search results; dedicated coupling graph view (future)
  → Prerequisite: P2k (condy metadata) for BSM-level edges

- ⏳ BootstrapMethods attribute scanning (condy-in-invokedynamic BSM args)              P4e
  → Novel analysis: scan ClassModel.attributes() → BootstrapMethodsAttribute directly,
    independent of method-body walking
  → Each BootstrapMethodEntry.arguments() can contain DynamicConstantDesc entries —
    condys nested as static arguments to invokedynamic call sites; these are never emitted
    as standalone ldc instructions and are completely invisible to method-body analysis
  → Emits new UsageType: CONDY_BOOTSTRAP_ARG — constant whose value is baked into a call
    site recipe rather than loaded directly; especially relevant for Kotlin and Groovy
  → For coupling analysis (P4d): two components sharing the same nested condy in their
    bootstrap args are coupled at the call-site-recipe level — the deepest form of hidden
    constant coupling detectable statically
  → Prerequisite: P2k (establishes condy handling patterns); P2j (calibration identifies
    which JVM languages actually emit this structure)

- ⏳ [DEFERRED] Constant split/merge analysis (detect refactoring patterns)             P4f
  → Detect when one constant splits into N or N merge into one across versions of the same project
  → Pure Java diff logic over data already in Postgres — no Solr involvement
  → Deferred: lower ROI compared to cross-component coupling (P4d); revisit after P4d ships
    and real usage patterns are observed
