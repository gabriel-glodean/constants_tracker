- ⏳ Classifier calibration: upload real-world JARs and inspect extracted constants      P2j
  → Goal: discover constant patterns not covered by existing 6 classifiers, especially
    CONSTANT_Dynamic (condy) cases emitted by non-javac compilers
  → Important feedback to verify during calibration:
      • A condy value is resolved by a bootstrap method at runtime and then cached per call site
      • Some condy values are not safely reducible to a static literal
      • Condy may be nested inside invokedynamic bootstrap data (including string concat recipes)
  → Candidate JARs (all on Maven Central):
      kotlin-stdlib.jar       — condy CONSTANT_VALUE bootstraps, lambda handles
      groovy-all.jar          — dynamic dispatch bootstrap args (Groovy 4.x)
      hibernate-core.jar      — SQL fragments, HQL strings, property keys (SQL classifier stress)
      jackson-databind.jar    — JSON keys, MIME types, date formats (multi-classifier)
      byte-buddy.jar          — JVM descriptor strings, method signature constants
      scala-library.jar       — Scala 3 condy for match type witnesses (different from Kotlin)
  → Expected new patterns to investigate:
      JVM_DESCRIPTOR          — strings matching `(L...;)V` descriptor format
      CONDY_BOOTSTRAP_ARG     — constants loaded exclusively as ConstantDynamic arguments
      BITMASK_OPERAND         — numeric ARITHMETIC_OPERANDs that are powers of 2 or hex masks
  → Outcome: document findings, add new CoreSemanticType entries or UsageType variants
    for patterns found in ≥ 2 libraries; discard one-offs

- ⏳ CONSTANT_Dynamic (condy) support in ConstantHandler                                P2k
  → Currently: ci.constantValue() instanceof DynamicConstantDesc flows into ObjectConstant
    and silently falls through AnalysisMerger — never emitted as a trackable usage
  → Fix (plain English):
      • Detect DynamicConstantDesc and try to resolve it only for known safe/deterministic bootstrap patterns
      • If resolved, emit a normal constant (NumericConstant/ObjectConstant) so existing analysis still works
      • If not resolved, do not drop it; emit a symbolic dynamic value (PHI-like placeholder) so it stays traceable
  → Symbolic value key should include bootstrap owner + name + descriptor + static args (+ cp index if needed)
    so each dynamic source is stable and uniquely identifiable
  → Store metadata on ConstantUsage for both resolved and unresolved condy:
      • bsmOwner, bsmName, bsmDescriptor, staticArgs
      • resolutionStatus: RESOLVED or SYMBOLIC
      • sourceContext: LDC or BOOTSTRAP_ARG
  → This metadata is the coupling signal for P4d
  → Do after P2j calibration (findings inform which BSMs are worth pattern-matching)

- ⏳ [TENTATIVE] Replace Solr with Postgres-native search (pg_trgm + pgvector)          P4c
  → Rationale: Solr brings no meaningful advantage for short string constants; BM25 is tuned
    for long documents; constants benefit more from type-aware matching
  → UnitConstantsStore SPI already isolates Solr behind an interface — swap is clean
  → Postgres replacement: pg_trgm GIN index for fuzzy search, fuzzystrmatch levenshtein()
    for edit-distance, value_num numeric column for type-aware numeric similarity
  → Eliminates: SolrService, SolrOutboxProcessor, SolrOutboxPayload, solr_outbox table,
    P0d (Resilience4j), P2h (parsePairs inconsistency), P4b (dead-letter replay),
    Solr container in docker-compose + k8s, managed-schema.xml, SolrJ dependency
  → Prerequisites: all planned search features (P3c) must be built first so all
    query patterns are known before rewriting the store layer

- ⏳ Cross-component constant coupling analysis                                          P4d
  → Core value proposition: detect when the same (or near-same) constant value appears in
    multiple independently-indexed projects/versions — a hidden coupling that code review
    cannot reliably catch across component boundaries
  → New query: GET /search/shared?value=...  → returns all projects+versions that contain
    this exact constant value, ranked by frequency; string and numeric types handled separately
  → Subsumes P3a (similar constant detection): near-same numeric range handled here via
    Postgres value_num column or in-memory post-fetch; Solr MLT within-project approach dropped
    as too narrow and Solr-specific
  → Coupling strength signal: if two components share a constant AND that constant appears in
    conditional logic (branch predicate via ARITHMETIC_OPERAND or comparison context), flag as
    HIGH_RISK coupling — a one-sided change silently alters branching behaviour in the other
  → Condy coupling edge (requires P2k): if two components reference the same DynamicConstantDesc
    BSM (same owner + name + static args), record a BSM-level coupling edge — stronger than
    value equality because it means both depend on the same code to produce the constant
  → UI: "Shared across projects" indicator in search results; dedicated coupling graph view (future)
  → Prerequisite: P2k (condy metadata) for BSM-level edges

- ⏳ BootstrapMethods attribute scanning (condy-in-invokedynamic BSM args)              P4e
  → Scan ClassModel.attributes() → BootstrapMethodsAttribute directly, not only method bodies
  → Reason: condy can be nested inside invokedynamic bootstrap arguments (including concat recipes),
    and those values may never appear as standalone ldc instructions
  → Emit new UsageType: CONDY_BOOTSTRAP_ARG for values found in bootstrap argument lists
  → Use the same symbolic key model as P2k for unresolved nested condy so they can still be compared
    across projects/versions
  → For coupling analysis (P4d): if two components share the same nested condy symbolic key,
    treat it as a strong hidden coupling signal at call-site recipe level
  → Prerequisite: P2k (establishes condy handling patterns); P2j (calibration identifies
    which JVM languages actually emit this structure)

- ⏳ [DEFERRED] Constant split/merge analysis (detect refactoring patterns)             P4f
  → Detect when one constant splits into N or N merge into one across versions of the same project
  → Pure Java diff logic over data already in Postgres — no Solr involvement
  → Deferred: lower ROI compared to cross-component coupling (P4d); revisit after P4d ships
    and real usage patterns are observed
